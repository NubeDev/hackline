# 2026-05-15 — Goal 14: drop MockApiClient — no-mock policy

The `MockApiClient` shipped with the original `hackline-ui` extraction
(now `@hackline/client`) and the `?mock=1` UI fast-path were
in-memory fixtures intended for "render the UI before the gateway is
up." Per project policy (and stated user preference) hackline's
testing and dev model is **real gateway, real Zenoh, no mocks** —
echoing `DOCS/CODEBASE-ANALYSIS.md` ("Loopback Zenoh router for E2E,
don't mock transport") and `DOCS/DEVELOPMENT.md` ("if it's wrong,
don't paper over it with mocks").

This goal makes that policy concrete: the mock surface is removed
from the package and the UI, and the docs name the policy out loud
so the next contributor doesn't reintroduce it.

## Design

**Why delete and not deprecate.** Keeping a `MockApiClient` around as
"deprecated, will be removed" is exactly the seam that lets it slip
back into use the next time the gateway is mid-refactor and somebody
wants the UI to render. The whole point of the policy is to make
running-without-the-real-gateway a friction the operator notices, so
they fix the gateway. Soft-deprecation does not produce that
friction.

**Why the no-token / down screens stay.** They aren't mocks — they
report real states ("the gateway is up but I have no bearer token",
"I cannot reach the gateway") that the operator has to act on. The
mock screen reported a fake state that happened to look like a happy
path; that's the failure mode we're cutting.

**Why update both READMEs.** Anyone landing in `@hackline/client` from
its README needs to know they will not find a built-in mock. Anyone
landing in `hackline-ui` from its README needs to know `?mock=1` is
gone and what to do instead.

**Out of scope.**

- Real integration tests against a loopback gateway. They want a
  fresh test harness (spin gateway in a child process, hit it via
  the package). Worth doing — separate goal so the diff stays
  focused on the removal.
- Replacing the dropped React-state machine with a router. Tempting
  while we're here, but unrelated.
- Touching `hackline-agent` or `DOCS/SCOPE.md` — both have unrelated
  in-flight work in the working tree.

## Outcome

The mock surface is gone end-to-end: removed from the package, the
UI shim, the boot path, and both READMEs. The no-mock policy is
named out loud in three places (`@hackline/client` index module-doc,
`@hackline/client` README, `hackline-ui` README) so the rule is
discoverable from any entry point.

Deleted / rewritten:

- `clients/hackline-ts/src/mock-client.ts` — deleted.
- `clients/hackline-ts/src/index.ts` — dropped `MockApiClient`
  re-export, added the no-mock policy comment.
- `clients/hackline-ts/src/client.ts` — dropped the `MockApiClient`
  reference from the interface module-doc.
- `ui/hackline-ui/src/main.tsx` — stripped `isMockRequested`,
  the `"mock"` Mode variant, the `MockBadge` component, the
  `?mock=1` mention in the ServerDown UX. Boot flow is now
  pure: probe → down / claim / no-token / ready.
- `ui/hackline-ui/src/lib/api/index.ts` — dropped the
  `MockApiClient` re-export.
- `ui/hackline-ui/README.md` — added a "No-mock policy" section
  with the rationale; replaced the `?mock=1` dev recipe with
  `make start`.
- `clients/hackline-ts/README.md` — added a "No mocks" section
  citing the same policy docs.

Verified:
- `pnpm -C clients/hackline-ts build` clean.
- `pnpm -C ui/hackline-ui typecheck` clean.
- `pnpm -C ui/hackline-ui build` succeeds, bundle ~258 KB / 78 KB gz
  (down ~4 KB / ~1 KB from before — fixtures deleted).
